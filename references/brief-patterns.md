# Design-Brief Composition

Read this after the `design` skill has classified the surface. The router owns surface doctrine; this file only shapes the commission.

## Prompt shape

Use natural headings for ordinary coding-agent prompts. Use descriptive XML tags only when long documents, examples, variable inputs, and instructions would otherwise blur together.

Include only sections that change the result:

1. **Commission** — the visual artefact or interface to create and the experience it should produce.
2. **Product truth** — user, workflow, content, capabilities, and semantic invariants.
3. **Visual direction** — concrete choices derived from the selected design route.
4. **Coverage** — required surfaces, realistic states/content, and viewports.
5. **Latitude** — presentation that may change and semantics that remain fixed.
6. **Working mode** — implement or create, render when tools permit, inspect, and iterate.
7. **Visual finish line** — observable qualities and completed states.
8. **Handoff** — changed surfaces, visual evidence, decisions, and deferred non-design issues.

## Route adapters

- **Public/brand:** turn the router's brand judgement into a complete visual narrative and conversion path.
- **Product/task:** turn the real task loop, hierarchy, density, provenance, status, editing, and responsive behaviour into the commission.
- **Audit/redesign:** name the visible defect and consequence; preserve sound identity while authorising visual restructuring where it is the defect.
- **Image-to-code:** label each screenshot, mock-up, Figma frame, or generated comp as an approved source of truth or inspiration. Approved sources prioritise fidelity in geometry, type, spacing, colour, imagery, and responsive transitions and skip direction exploration.
- **Design system:** commission the required tokens, typography, colour, spacing, motion, primitives, variants, states, documentation artefact, and representative applied surfaces; leave package-wide migration and regression assurance to closure.

## Direction decision

Replace vague labels such as “premium”, “modern”, or “clean” with observable choices. If those choices are materially open, present a small set of genuinely different directions, recommend one, and ask one blocking question. State that no response means the recommendation is selected and implementation continues. Inspiration can inform options; an approved source of truth constrains them.

## Mixed-programme allocation

```text
Preparation: establish the functional contract, working preview, relevant frontend files, representative data/states, screenshots, assets, and constraints.

Visual production: use that context to create and implement the selected direction. Render the required states and viewports when tools permit, inspect, and iterate. Return visual evidence and design decisions. Do not request automated assurance or release evidence.

Engineering closure: integrate and own tests, builds, CI, formal audits, backend compatibility, deployment, and production proof.
```
